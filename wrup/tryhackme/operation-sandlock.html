operator brief

Introduction

Operation SandLock is a TryHackMe CTF room designed to test analysis, reverse engineering, and forensics skills.

The story begins when the security team Error.exe discovers a compromised workstation in the heart of Amman. After collecting the evidence, the team recovers several suspicious files connected to an attacker known as The Sandman.

Each file appears to protect one part of a larger secret.

The Sandman left one message behind:

Four locks. One truth.

The goal of this room is to follow the clues, analyze the files, solve each lock in order, and uncover the final secret.

Task 2 — Crypto Analysis: The Encrypted Note

Challenge Overview

The team’s crypto analyst, Laith, found a suspicious encrypted note. The file contained a strange hex string. Nothing was readable at first, but The Sandman left a clue:

The first lock listens to the city where it all began.

Since the story mentioned that the compromised workstation was found in Amman, the clue pointed back to the city name.

Initial Analysis

The encrypted note contained a hex-encoded value. Since the clue referenced the city where everything began, I used amman as the key.

The city name was important because it was not just part of the story; it was also the hint needed to unlock the first encrypted message.

Task 3 — Reverse Engineering: The Gatekeeper of the Sand

Challenge Overview

Naim, the reverse engineer of the team, takes over this part of the operation. The provided binary asks for an access key when executed:

./gatekeeper
Enter the access key:

At first, the key recovered from the previous task, desert, may look like the correct password. However, the binary does not accept that value directly.

Running File Analysis

The binary name started with a dash:

-1782436588904.gatekeeper

When I tried to run file normally, Linux treated the filename as command options:

file -1782436588904.gatekeeper

To fix that, I used -- before the filename or referenced the file with ./:

file -- -1782436588904.gatekeeper
file ./-1782436588904.gatekeeper

Recommended Tool

The correct tool to analyze the binary is:

Ghidra

Ghidra allows us to inspect the binary, view decompiled code, and understand how the input validation works.

Reverse Engineering in Ghidra

After importing the binary into Ghidra, I inspected the decompiled function responsible for checking the user input. The binary contained encoded strings and a decoding function.

The important part was that the program copied encoded data into memory, called another function, and then compared the decoded result with the user input using strcmp().

XOR Decoding Routine

The decoding function used a simple XOR operation. It looped over the encoded bytes and XORed each byte with:

0x37

This means the hidden access key was not stored directly in plaintext. It was decoded at runtime before being compared to the input.

After decoding the hidden value, the correct access key was:

sands_everywhere

Proof of Concept

./-1782436588904.gatekeeper
sands_everywhere

The binary accepted the key and revealed the second flag and the second key fragment.

Task 4 — Forensics: Hell Sands / The Vault

Challenge Overview

After completing the previous stages, the team now had two separate key fragments:

KEY_PART_1 = desert
KEY_PART_2 = -rabbit

The clue says:

Truth waits inside, but do not believe what you see.

Combining the Key Fragments

The password was not a new value. It had to be built by combining the two key fragments from the previous tasks:

desert-rabbit

I used this password to extract the archive:

unzip secret-sands-1782576720114.zip
Password: desert-rabbit

The archive extracted a file named 222.jpg.

Checking the Extracted File

Although the extracted file had a .jpg extension, I checked it using the file command:

file 222.jpg
222.jpg: data

This means the file was not recognized as a valid JPEG image. Since the extension suggested it should be an image, I suspected that the file header was corrupted.

Repairing the JPEG Header

I opened the file using a hex editor:

hexeditor 222.jpg

A valid JPEG file commonly starts with:

FF D8 FF E0

I replaced the corrupted first four bytes with FF D8 FF E0. After saving the file, it was recognized as a JPEG image.

Viewing the Repaired Image

After repairing the header, I opened the image and found a flag displayed inside it.

Task 5 — The Final Vault

Challenge Overview

The previous task revealed a hidden hash inside the repaired image:

VAULT_HASH=8621ffdbc5698829397d97767ac13db3

The goal was to crack the hash, use the recovered value as the password, open the final vault, and retrieve the final flag.

Cracking the Hash with Hashcat

The hash is 32 hexadecimal characters long, which strongly suggests MD5. I saved it into a file and cracked it using Hashcat mode 0:

echo -n "8621ffdbc5698829397d97767ac13db3" > hash.txt
hashcat -m 0 -a 0 hash.txt /usr/share/wordlists/rockyou.txt
hashcat -m 0 --show hash.txt

The output was:

8621ffdbc5698829397d97767ac13db3:dragon

The recovered password was:

dragon

Unlocking the Final Vault

I used the cracked password to extract the final vault archive:

unzip vault-1782576742090.zip
Password: dragon
cat final.txt

The file contained the final flag.

after action report

Conclusion

Operation SandLock is complete. The investigation started with an encrypted note, moved through a protected binary, continued with a locked vault, and ended with hidden evidence inside the final extracted files.

  • Laith used cryptography skills to decode the encrypted note and recover the first key fragment.
  • Naim used reverse engineering skills to analyze the gatekeeper binary and recover the second key fragment.
  • Ahmad used forensics skills to unlock the vault, repair hidden evidence, and crack the final password.

Throughout this room, several important CTF skills were practiced: hex decoding, Vigenère decoding, Base64 decoding, Base32 decoding, binary analysis, reverse engineering, XOR decoding, password chaining, file inspection, hex editing, hash cracking, and hidden data extraction.

Three analysts. Four locks. One truth.

Although the team unlocked the final vault, the case may not be fully closed. The Sandman is still unknown, and the evidence points to a larger operation.

Next time, the mission will not only be to open his locks. It will be to find him, track him, and finally catch him.
