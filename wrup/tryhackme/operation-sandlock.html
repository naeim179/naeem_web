Overview

Operation SandLock is a TryHackMe CTF room built around analysis, reverse engineering, and forensics. The story begins when the security team Error.exe discovers a compromised workstation in Amman and recovers suspicious files connected to an attacker known as The Sandman.

The Sandman left one message behind:

Four locks. One truth.

The goal was to follow the clues, analyze each artifact, solve the locks in order, and uncover the final secret.

solution notes

Walkthrough

Task 1 — Introduction

The first task introduces the storyline and explains the objective of the challenge. There is no technical answer required here; it sets the context for the rest of the investigation.

Task 2 — Crypto Analysis: The Encrypted Note

The crypto analyst found a suspicious encrypted note containing a hex-encoded value. The clue said:

The first lock listens to the city where it all began.

Since the story started in Amman, I used amman as the key.

The CyberChef recipe was:

  1. From Hex
  2. Vigenère Encode using the key amman
  3. From Base64

The decoded output revealed the first redacted flag and a Base32-looking value. Decoding that value with From Base32 recovered the first key fragment:

KEY_PART_1 = desert

Task 3 — Reverse Engineering: The Gatekeeper of the Sand

The provided binary asked for an access key:

./gatekeeper
Enter the access key:

The first fragment, desert, was not accepted directly. This showed that the previous key was only part of a longer chain.

The binary filename started with a dash, so running tools against it required either -- or a relative path prefix:

file -- -1782436588904.gatekeeper
file ./-1782436588904.gatekeeper

I used Ghidra to inspect the decompiled validation logic. The program copied encoded bytes into memory, decoded them at runtime, and compared the decoded result to the user input with strcmp().

The decoding routine XORed each encoded byte with:

0x37

After decoding the hidden value, the correct access key was:

sands_everywhere

Entering that key revealed the second redacted flag and the second key fragment:

KEY_PART_2 = -rabbit

Task 4 — Forensics: Hell Sands / The Vault

After Task 2 and Task 3, the chain had two fragments:

KEY_PART_1 = desert
KEY_PART_2 = -rabbit

Combining them produced the archive password:

desert-rabbit

I used that password to extract the archive:

unzip secret-sands-1782576720114.zip

The extracted file was named 222.jpg, but file reported it as raw data:

file 222.jpg
222.jpg: data

Because the extension suggested a JPEG image, I inspected the file header in a hex editor. A common JPEG header begins with:

FF D8 FF E0

After replacing the corrupted first four bytes with the valid JPEG magic bytes, the image opened successfully and displayed a redacted flag.

The clue warned not to believe only what was visible, so I inspected the image with strings and found a hidden hash:

strings 222.jpg
VAULT_HASH=8621ffdbc5698829397d97767ac13db3

Task 5 — The Final Vault

The hidden hash was 32 hexadecimal characters, which strongly suggested MD5:

8621ffdbc5698829397d97767ac13db3

I saved the hash and cracked it with Hashcat using MD5 mode -m 0:

echo -n "8621ffdbc5698829397d97767ac13db3" > hash.txt
hashcat -m 0 -a 0 hash.txt /usr/share/wordlists/rockyou.txt
hashcat -m 0 --show hash.txt

Hashcat recovered the password:

8621ffdbc5698829397d97767ac13db3:dragon

I used dragon to unlock the final vault archive:

unzip vault-1782576742090.zip
cat final.txt

The final file contained the final redacted flag.

Task 6 — Conclusion

Operation SandLock ends after four chained locks. Laith handled the crypto analysis, Naim handled the reverse engineering, and Ahmad handled the forensics and final vault work. Each stage revealed a piece of the larger truth.

after action report

Lessons Learned

  • Story clues can be technical hints, not just narrative decoration.
  • When a recovered value fails, treat it as a chain fragment instead of the final answer.
  • Filenames beginning with - should be handled with -- or ./ to avoid option parsing issues.
  • Binary secrets are often decoded at runtime; checking decompiled logic and string comparisons can reveal the validation path.
  • File extensions cannot be trusted. Always verify evidence with file, strings, and a hex editor when needed.
  • Hidden hashes inside repaired evidence can lead to the final vault stage.

The Sandman left four locks behind, and each lock revealed part of the truth. Although the final vault was opened, the attacker remains unknown. The next mission will not only be to open the locks, but to track The Sandman and finally catch him.
