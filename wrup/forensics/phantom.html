Walkthrough

Phantom

Forensics / Network Analysis

boroCTF{M1nd_th3_G4P}

1. Initial Inspection

After extracting the challenge archive, we found two files:

unzip Phantom.zip
ls player_dist
Output:
network_map.html
phantom.pcap
The network_map.html file shows a network topology for Apex Energy. The important host in the diagram is:

Database Server: 10.0.50.100

This immediately gives us a target IP to focus on inside the packet capture.

2. Analyzing the PCAP

The file phantom.pcap contains a huge number of packets, most of which are junk traffic.

When looking at the traffic, we notice many packets containing:

junk_traffic

So the main goal is to find the packets that are different from the noise.

After checking the flows, one suspicious flow stands out:

10.0.50.100:54321 -> 198.51.100.22:443

This flow appears at the very end of the capture and contains 21 packets.

The interesting part is that these packets do not contain normal payload data. Their payload length is:

0 bytes

So the data is not hidden in the TCP payload.

3. Finding the Hidden Data

Since there was no payload, I checked the TCP header/options.

The suspicious packets all contained a TCP Timestamp option.

The extracted TCP timestamp values were:

72 69 88 69 105 126 108 81 103 27 68 78 117 94 66 25 117 109 30 122 87

At first, these values do not directly decode to readable ASCII.

So I tried XOR decoding. Using XOR key:

0x2a

The values decode correctly.

Example:

72 ^ 0x2a = 98 -> 'b'

69 ^ 0x2a = 111 -> 'o'

88 ^ 0x2a = 114 -> 'r'

69 ^ 0x2a = 111 -> 'o'

This starts forming:

boro

So the timestamp values were the hidden message, XOR-encrypted with 0x2a.

4. Extraction Script

ts_values = [

72, 69, 88, 69, 105, 126, 108,

81, 103, 27, 68, 78, 117, 94,

66, 25, 117, 109, 30, 122, 87

]
key = 0x2a
flag = ''.join(chr(x ^ key) for x in ts_values)
print(flag)
Output:
boroCTF{M1nd_th3_G4P}

5. Flag

boroCTF{M1nd_th3_G4P}

Summary

The challenge was hiding data in a covert channel. Most of the PCAP was junk traffic, but the network map pointed us toward the internal database server 10.0.50.100. The suspicious connection from this host had no payload, so the real data was hidden inside the TCP Timestamp option. Extracting those timestamp values and XORing them with 0x2a revealed the flag.

after action report

Lessons Learned

  • Covert channels can hide in protocol metadata, not payloads.
  • Provided diagrams can dramatically narrow PCAP analysis.
  • A quick XOR brute/check can validate suspicious byte streams.
