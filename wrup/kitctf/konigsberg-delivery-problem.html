Walkthrough

Königsberg Delivery Problem — GPNCTF Writeup

Challenge Information

Category: Reversing Points: 100 Challenge Name: Königsberg Delivery Problem

Description

> Euler is delivering pizza in Königsberg, but his route planning skills are questionable. Can you help him?

The title immediately hints at the famous Seven Bridges of Königsberg problem, which led Euler to the creation of graph theory and Eulerian paths.

Initial Analysis

Running file on the binary:

file cartographer

Output:

ELF 64-bit LSB executable
not stripped

The binary was not stripped, making reversing significantly easier.

Opening the binary in Ghidra revealed only a few interesting functions:

main
cfg
check_instance

The most suspicious function was cfg(), which was extremely large and contained hundreds of jump-table branches.

Analyzing main()

The decompiled main() looked unusual:

scanf(fmt_0,   &local_102);
scanf(fmt_1,   &local_101);
scanf(fmt_2,   &local_100);
...
scanf(fmt_249, &local_9);

cfg(&local_102);

The program performs 250 separate scanf calls and then passes the resulting buffer to cfg().

This tells us:

  1. The input length is fixed.
  2. The challenge expects exactly 250 values.
  3. The entire validation logic is hidden inside cfg().

Understanding cfg()

The cfg() function is enormous.

Instead of performing ordinary checks, it behaves like a giant state machine:

switch(state) {
    case 0:
        ...
    case 1:
        ...
    case 2:
        ...
}

Every input byte causes a transition from one node to another.

The function name itself is a clue:

cfg = Control Flow Graph

The challenge is literally asking us to traverse a graph.

The Graph Interpretation

After reconstructing the state transitions, the structure becomes a graph:

node0 -> node2
node2 -> node3
node3 -> node4
...

The challenge title now makes sense.

Euler's Königsberg problem is a graph traversal problem.

The validator expects a specific walk through the graph.

Recovering the Correct Path

By extracting all transitions and solving the graph, we obtain a sequence of node IDs that traverses the graph correctly and eventually reaches the success state.

The recovered path contains 250 values.

Generating the Payload

A simple Python script can generate the required input:

vals = [
0,0,0,2,3,0,3,4,3,3,5,6,3,6,5,6,
# ... 250 values total ...
78,90,102,98,89,115,111,98,112,127
]

print(";".join(map(str, vals)) + ";")

The binary expects values separated by semicolons.

Example:

0;0;0;2;3;0;...

Local Verification

Running locally:

./cartographer < payload.txt

produces:

Congratulations!

or directly prints the flag when the local flag file exists.

Remote Exploitation

Connecting to the remote service:

ncat --ssl deep-fried-paella-beside-toasted-rosemary-vdis.gpn24.ctf.kitctf.de 443

and sending the generated payload returns:

GPNCTF{say_Euler_ThE_OwL_Owl5_1N_KöNIgs8erg_10_tImES_FAs7!}
GPNCTF{say_Euler_ThE_OwL_Owl5_1N_KöNIgs8erg_10_tImES_FAs7!}

Takeaways

This challenge disguises a graph traversal problem as a massive control-flow graph.

Key observations:

  1. cfg() literally stands for Control Flow Graph.
  2. The challenge title references Euler and Königsberg.
  3. The validator is implemented as a huge state machine.
  4. Solving the graph yields the exact sequence of 250 node transitions required to reach the success state.

The challenge is a clever combination of reversing and graph theory, where recognizing the Euler/Königsberg hint is the key to understanding the intended solution path.

after action report

Lessons Learned

  • Challenge titles can reveal the mathematical model.
  • A large switch/cfg can represent a graph rather than ordinary checks.
  • Generating the validator input is cleaner than manually entering values.
