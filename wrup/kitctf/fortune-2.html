GPN CTF 2024 – Fortune 2 Writeup

Challenge Overview

We are given a remote service:

ncat --ssl charred-chorizo-on-crispy-hollandaise-7na5.gpn24.ctf.kitctf.de 443

The challenge source contains an NTRU-like encryption scheme implemented with polynomial arithmetic. At first glance it looks like a lattice/NTRU problem that would require recovering secret polynomials using LLL or other advanced techniques.

However, the vulnerability is much simpler.

Source Analysis

The encryption function is:

def encrypt(self,message):
    r = self.sampleShort()
    t = self.pk * r
    return (t).rescale(p)+message

where:

p = 3

The ciphertext is therefore:

[ c = \text{rescale}(t) + m ]

The important observation is how rescale() works.

Inside fortuneUtils2.py:

def rescale(self,p):
    return Poly([lift_centered(i,p) for i in self.coeffs], self.N)

and:

def lift_centered(a,p):
    a = a % p
    if a > p//2:
        return a-p
    return a

For p = 3, every coefficient of rescale(t) is reduced into:

[ {-1,0,1} ]

and specifically:

[ \text{rescale}(t) \equiv t \pmod 3 ]

Since the public key construction guarantees that the encrypted noise term is divisible by 3 modulo the ring, we obtain:

[ \text{rescale}(t) \equiv 0 \pmod 3 ]

Therefore:

[ c \equiv m \pmod 3 ]

which means every ciphertext coefficient directly reveals the corresponding message coefficient.

Message Encoding

The challenge encodes letters as:

A -> -1
B -> 1
C -> 0

Reducing modulo 3:

SymbolValuemod 3
A-12
B11
C00

So we can recover the plaintext using:

c[i] % 3

and mapping:

0 -> C
1 -> B
2 -> A

Exploit

After connecting to the server we receive a ciphertext vector:

c=[...]

We simply compute:

guess = "".join(mapping[x % 3] for x in c)

and send it back.

Solver Script

import socket
import ssl
import re
import ast

HOST = "charred-chorizo-on-crispy-hollandaise-7na5.gpn24.ctf.kitctf.de"
PORT = 443

ctx = ssl.create_default_context()

with ctx.wrap_socket(socket.socket(), server_hostname=HOST) as s:
    s.connect((HOST, PORT))

    data = b""
    while b"Give me the message:" not in data:
        data += s.recv(8192)

    txt = data.decode()

    c_txt = re.search(r"c=\s*(\[[^\]]+\])", txt).group(1)
    c = ast.literal_eval(c_txt)

    mapping = {
        0: "C",
        1: "B",
        2: "A"
    }

    guess = "".join(mapping[x % 3] for x in c)

    print("Recovered message:")
    print(guess)

    s.sendall((guess + "\n").encode())

    print(s.recv(8192).decode())

Running the script immediately recovers the plaintext and the server returns:

Takeaway

The challenge is designed to look like an NTRU/lattice cryptography problem, but the ciphertext leaks the plaintext directly modulo 3. Instead of attacking the lattice structure, we exploit the fact that:

[ c \equiv m \pmod 3 ]

allowing complete message recovery with a simple modular reduction. This is a classic example of a cryptographic implementation leaking information through an incorrect modulus handling step.

Lessons Learned

  • A complicated cryptosystem can fail through a simple encoding leak.
  • Always check what the ciphertext reveals modulo small parameters.
  • Source-level mistakes can avoid the need for heavy lattice attacks.
