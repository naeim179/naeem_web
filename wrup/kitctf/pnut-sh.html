Walkthrough

Writeup: Pnut-sh (GPNCTF)

Challenge Overview

The challenge provides a custom C interpreter implemented in a Shell script (pnut-sh.sh). The server accepts a single line of C code, parses it, and executes it within the shell environment. The goal is to retrieve the flag located in /flag.

Analysis

  • Environment: The system includes a binary /usr/bin/nl with the setuid bit set (chmod u+s /usr/bin/nl), allowing any user to read restricted files as root.
  • Limitation: The interpreter (pnut-sh.sh) does not support the system() function natively, as attempting to call it results in a command-not-found error (e.g., _system: command not found).
  • Capability: The interpreter supports standard C file I/O operations such as fopen, fgetc, and printf.

Exploitation Strategy

Since the goal is to read the /flag file and the system() function is unavailable, we can use standard C file manipulation functions. By opening the file with fopen and iterating through its content with fgetc, we can stream the output of the flag to the standard output.

Payload:

int main(){int f=fopen("/flag","r");while(1){int c=fgetc(f);if(c==-1)break;printf("%c",c);}return 0;}

after action report

Lessons Learned

  • When command execution is blocked, direct file APIs may still work.
  • Minimal single-line payloads are useful for restricted interpreters.
  • Custom interpreters often expose more primitives than intended.
